# Phase 5 v5: Realistic Prompt Experiment

**Date**: 2026-04-15
**Status**: Designed

## Motivation

Phases 5 v1-v4 all failed to differentiate Agent A (no graph) from
Agent B (with graph) on recall. The root cause: **every previous
experiment gave both agents explicit, fully-specified instructions**.
When the prompt says "update every call site" or "mark every
transitively affected file," the agent knows exactly what to find —
the graph just makes it faster.

In reality, users don't give explicit instructions. They say "I
changed X, make sure nothing is broken" or "rename this file." The
graph's value is not just answering queries — it's **prompting the
agent to ask the right questions**. The CLAUDE.md rules injected by
`kb-graph init` tell the agent to read KB_INDEX.md and query the
graph before multi-file changes. Without those rules, the agent may
not realize there's a blast radius to check.

## Hypothesis

When prompts are vague and realistic (matching how users actually
talk to coding agents), Agent B will achieve significantly higher
recall than Agent A because:

1. Agent B's CLAUDE.md rules **nudge it to check for blast radius**
2. KB_INDEX.md and `kb-graph traverse` **give it a systematic
   checklist** of affected files
3. Agent A must independently realize that changes have downstream
   consequences AND manually trace them

## Design

## Scenario A: "Fix this thing" — Discover downstream breakage

### Fixture

Reuse Experiment B's 80-file Python data pipeline fixture with one
modification: `src/core/transform.py` is pre-modified so that
`apply_transform()` already has the new signature:

```python
def apply_transform(data, config, mode: str = "strict"):
```

All 6 call sites across 5 files still use the old `strict=True` /
`strict=False` syntax. The project is in a broken state but the agent
isn't told which files are broken.

Agent B's project additionally has:
- `CLAUDE.md` with kb-graph rules (read KB_INDEX.md first, query
  graph before multi-file changes)
- `KB_INDEX.md` generated by `kb-graph init`
- `.claude/skills/check_graph/SKILL.md`

### Prompts

**Prompt A** (no graph):

```
I updated apply_transform() in src/core/transform.py — the strict
parameter is now called mode (takes "strict" or "lenient" instead of
True/False). Can you make sure I didn't break anything?
```

**Prompt B** (with graph):

```
Read CLAUDE.md first. I updated apply_transform() in
src/core/transform.py — the strict parameter is now called mode
(takes "strict" or "lenient" instead of True/False). Can you make
sure I didn't break anything?
```

### Scoring

Same regex-based call-site scoring from Experiment B v4. For each of
the 6 expected fixes, check if:
- The old pattern (`strict=True` / `strict=False`) is gone
- The new pattern (`mode="strict"` / `mode="lenient"`) is present

**Expected fixes** (6 call sites in 5 files):

| File | Old | New |
|------|-----|-----|
| `src/preprocessing/normalize.py` | `strict=True` | `mode="strict"` |
| `src/analysis/aggregate.py` | `strict=False` | `mode="lenient"` |
| `src/pipeline/run.py` | `strict=True` | `mode="strict"` |
| `tests/test_transform.py` (line 9) | `strict=True` | `mode="strict"` |
| `tests/test_transform.py` (line 14) | `strict=False` | `mode="lenient"` |
| `tests/test_pipeline.py` | `strict=False` | `mode="lenient"` |

Metrics: recall, precision, false positives, strategy detection.

### Expected Outcome

- **Agent A**: 30-80% recall. Might grep for `strict=` and find some
  call sites, but the vague prompt ("make sure I didn't break
  anything") doesn't explicitly tell it to search for call sites. It
  may read `transform.py`, confirm the change looks correct, and
  report "looks good" without checking callers.
- **Agent B**: 90-100% recall. CLAUDE.md rules tell it to read
  KB_INDEX.md and check the graph. KB_INDEX.md lists every file that
  imports `transform.py`. The agent checks each one systematically.

## Scenario B: "Something is broken" — Root cause vs symptom

### Fixture

Same 80-file Python fixture as Scenario A (transform.py pre-modified,
call sites still broken). Additionally includes a `test_output.txt`
file simulating a test failure:

```
$ python -m pytest tests/ -x
============================================ FAILURES =============================================
_________________________________ test_strict_mode _________________________________

    def test_strict_mode():
        config = TransformConfig(normalize=True)
>       result = apply_transform([], config, strict=True)
E       TypeError: apply_transform() got an unexpected keyword argument 'strict'

tests/test_transform.py:9: TypeError
======================================= short test summary =======================================
FAILED tests/test_transform.py::test_strict_mode - TypeError: apply_transform() got an unexpected keyword argument 'strict'
======================================= 1 failed in 0.42s ========================================
```

The traceback points at one test file. The question: does the agent
fix just the test (symptom) or trace back to the signature change and
fix all 6 call sites (root cause)?

### Prompts

**Prompt A** (no graph):

```
The tests are failing — see test_output.txt for the traceback. Can
you fix whatever is broken?
```

**Prompt B** (with graph):

```
Read CLAUDE.md first. The tests are failing — see test_output.txt
for the traceback. Can you fix whatever is broken?
```

### Scoring

Same 6 call-site regex scoring as Scenario A. Additionally, a
secondary classification:

- **Symptom-only fix**: agent modified only `tests/test_transform.py`
  (and possibly `tests/test_pipeline.py`) — max 3/6 recall
- **Root cause fix**: agent found and fixed call sites in source files
  (`normalize.py`, `aggregate.py`, `run.py`) AND test files — 6/6
  recall
- **Revert fix**: agent changed `transform.py` back to `strict=True`
  — 0/6 recall (undid the intended change). Detected by checking if
  `transform.py` still has the `mode` parameter.

The revert case is important: a reasonable agent might "fix" the
tests by reverting the signature change. We score this as 0/6 because
the user intended the signature change. Detection: check whether
`transform.py` still contains `mode: str = "strict"` after the agent
runs. If not (revert detected), skip call-site scoring entirely and
classify as "revert" — otherwise the 6 call sites still have `strict=`
which would spuriously look "correct" since the old signature is back.

Metrics: recall, precision, fix classification (symptom/root-cause/
revert), strategy detection.

### Expected Outcome

- **Agent A**: 30-50% recall. Reads `test_output.txt`, sees the error
  in `test_transform.py`, fixes that file. Might fix `test_pipeline.py`
  too if it notices similar patterns. Unlikely to proactively check
  non-test source files. May also attempt a revert fix.
- **Agent B**: 90-100% recall. CLAUDE.md rules lead it to KB_INDEX.md,
  which shows all files importing `transform.py`. The agent checks
  each one and fixes all 6 call sites.

## Scenario C: "Rename a file" — Discover broken references

### Fixture

Reuse Experiment A's 35-file wiki-link documentation fixture. No
pre-modification — the fixture is intact. The target file
`docs/design/02-processing-engine.md` exists and is referenced by 7
other files:

- 6 files with `[[02-processing-engine]]` wiki-links (depth-1)
- 1 config file with a path reference (`configs/platform_config.yaml`)

Agent B's project additionally has CLAUDE.md, KB_INDEX.md, and the
check_graph skill installed.

### Prompts

**Prompt A** (no graph):

```
Rename docs/design/02-processing-engine.md to
docs/design/02-data-processing.md. Update the content to reflect the
new name.
```

**Prompt B** (with graph):

```
Read CLAUDE.md first. Rename docs/design/02-processing-engine.md to
docs/design/02-data-processing.md. Update the content to reflect the
new name.
```

### Scoring

Two layers:

1. **File renamed** (boolean): old path gone, new path exists with
   non-empty content.

2. **References updated**: check 7 files for updated references.

   | File | Old reference | New reference |
   |------|--------------|---------------|
   | `docs/design/00-platform-overview.md` | `[[02-processing-engine]]` | `[[02-data-processing]]` |
   | `docs/design/01-data-ingestion.md` | `[[02-processing-engine]]` | `[[02-data-processing]]` |
   | `docs/design/05-scheduling.md` | `[[02-processing-engine]]` | `[[02-data-processing]]` |
   | `docs/design/10-architecture.md` | `[[02-processing-engine]]` | `[[02-data-processing]]` |
   | `docs/reference/design-notes.md` | `[[02-processing-engine]]` | `[[02-data-processing]]` |
   | `configs/platform_config.yaml` | `docs/design/02-processing-engine.md` | `docs/design/02-data-processing.md` |

   We do NOT score depth-2 files — those link to depth-1 files, not
   to the renamed target. Only direct references need updating.

   Recall = references updated / 6 total direct references (5
   wiki-links + 1 config path).

Metrics: file renamed (bool), reference recall, reference precision,
config path updated (bool), strategy detection.

### Expected Outcome

- **Agent A**: File renamed, 0-30% reference recall. "Rename and
  update content" sounds like a single-file task. The agent renames
  the file, maybe updates its title/heading, and reports done. It has
  no reason to suspect 6 other files contain wiki-links to the old
  name unless it proactively searches.
- **Agent B**: File renamed, 80-100% reference recall. CLAUDE.md
  rules say to query the graph before changes. `kb-graph traverse`
  reveals all 6 wiki-link dependents. The agent updates each one.

## Strategy Detection

For all three scenarios, analyze each agent's transcript to classify
its approach:

```python
def detect_strategy(transcript):
    text = transcript.lower()
    return {
        "used_graph": any(k in text for k in [
            "kb_index", "kb-graph", "traverse", "blast radius",
            "knowledge graph"
        ]),
        "used_grep": any(k in text for k in [
            "grep", "rg ", "ripgrep", "search for",
            "searching for"
        ]),
        "read_claude_md": "claude.md" in text,
        "checked_imports": any(k in text for k in [
            "who imports", "which files import", "files that import",
            "check.*import", "importing transform"
        ]),
    }
```

Include strategy data in results JSON. This lets us distinguish
"Agent B got 100% because it used the graph" from "Agent B got 100%
because it happened to grep" — even when recall is identical.

## Dry-Run Validation

`--dry-run` mode for each scenario:

- **Scenario A**: Build fixture, verify transform.py has new
  signature, verify all 6 call sites still have old syntax, run
  kb-graph init on project B, verify KB_INDEX.md lists transform.py
  importers.
- **Scenario B**: Same as A, plus verify `test_output.txt` exists
  with the traceback content.
- **Scenario C**: Build fixture, verify target file exists, verify
  all 6 wiki-link references exist in depth-1 files, run kb-graph
  init on project B, verify traverse output matches expected
  references.

## Script Structure

```
test_experiment_v5.py
  ├── FIXTURE_FILES_CODE = {...}        # 80-file Python fixture (from Exp B)
  ├── FIXTURE_FILES_DOCS = {...}        # 35-file wiki-link fixture (from Exp A)
  ├── EXPECTED_FIXES_A = [...]          # 6 call-site fixes (Scenarios A, B)
  ├── EXPECTED_REFS_C = [...]           # 6 wiki-link references (Scenario C)
  ├── TEST_OUTPUT_TXT = "..."           # Fake traceback (Scenario B)
  │
  ├── setup_scenario_a(tmpdir) -> (dir_a, dir_b)
  ├── setup_scenario_b(tmpdir) -> (dir_a, dir_b)
  ├── setup_scenario_c(tmpdir) -> (dir_a, dir_b)
  │
  ├── score_scenario_a(project_dir) -> dict
  ├── score_scenario_b(project_dir) -> dict
  ├── score_scenario_c(project_dir) -> dict
  │
  ├── detect_strategy(transcript) -> dict
  ├── run_agent(project_dir, prompt, ...) -> (stdout, stderr, rc, elapsed)
  ├── run_trial(scenario, trial_num, ...) -> dict
  ├── print_summary(scenario, results) -> dict
  └── main()
```

Scenario A and B share the same fixture (80-file Python project) with
minor differences (Scenario B adds `test_output.txt`). Scenario C
uses the 35-file docs fixture.

Setup functions:
1. Create tmpdir with two copies of the fixture (dir_a, dir_b)
2. For Scenario A/B: pre-modify `transform.py` in both copies
3. For Scenario B: add `test_output.txt` to both copies
4. Run `kb-graph init` on dir_b (Agent B's project)
5. Verify fixture integrity

## What Success Looks Like

The experiment succeeds if **at least one scenario** shows a
statistically significant recall gap between Agent A and Agent B.
Specifically:

- Agent B recall >= 90% AND Agent A recall <= 60% in that scenario
- OR Agent B achieves perfect runs in >= 80% of trials while Agent A
  achieves perfect runs in <= 30%

If all three scenarios show differentiation, that's strong evidence
the graph + CLAUDE.md rules provide measurable value for realistic
coding tasks.

If no scenario differentiates: the graph's value is speed only, and
phoam_paint should be positioned as an efficiency tool, not a
correctness tool. That's still valuable — 2-2.5x speedup is real —
but it changes the pitch.

---

## Relationship to Phase 6

This experiment is independent of Phase 6. It uses existing kb-graph
features (traverse, KB_INDEX.md, CLAUDE.md rules). Phase 6 adds
richer code intelligence (AST parsing, call-site indexing, R/shell
parsers) which could improve Agent B's performance in future
experiments but is not required for v5.

If v5 differentiates: Phase 6 is incremental improvement on proven
value.
If v5 fails: Phase 6 features (especially 6.2 call-site indexing)
might be what tips the balance, and should be tested in a v6
experiment.

# Scripts Reference

Generate a `scriptReferences.md` file that catalogs production scripts in a project with clickable file links.

## Invocation

Parse the argument after `/scripts-reference`:

| Invocation | Behavior |
|---|---|
| `/scripts-reference` | Prompt user for the production scripts directory |
| `/scripts-reference path/to/scripts` | Use the given directory directly |
| `/scripts-reference path/to/scripts -o output.md` | Use the given directory and custom output filename |

If no output path is specified, default to `scriptReferences.md` in the project root.

## Workflow

### 1. Identify the scripts directory

Get the **production scripts directory** â€” the folder containing scripts that are deployed, run in production, or used as part of the live system. This is NOT:

- Test scripts or test helpers
- Local dev-only scripts (setup, seeding, mocks)
- CI/CD pipeline configs (unless they are runnable scripts)
- One-off throwaway scripts

If the user did not provide a path, ask:

> What directory contains your production scripts? (e.g. `scripts/`, `bin/`, `src/scripts/`)

Accept the answer and resolve it to an absolute path.

### 2. Scan the directory

Recursively scan the target directory for script files. Include files matching common script extensions and shebangs:

- Shell: `.sh`, `.bash`, `.zsh`
- Python: `.py`
- R: `.R`, `.Rmd`
- Perl: `.pl`
- Ruby: `.rb`
- JavaScript/TypeScript: `.js`, `.ts` (only if they look like CLI scripts, not library modules)
- Any file with a shebang (`#!/...`) in the first line

Skip files that look like tests, configs, or non-script artifacts:
- Files starting with `test_`, ending with `_test.*` or `.test.*`
- `__pycache__/`, `node_modules/`, `.git/`
- `__init__.py`, `conftest.py`, `setup.py`, `setup.cfg`

### 3. Determine namespace and description

For each script found:

- **Name**: The filename (e.g. `run_alignment.sh`)
- **Namespace**: The relative directory path from the scripts root (e.g. `pipeline/alignment`). Use `.` for scripts directly in the root directory.
- **Description**: Read the file and extract a description from:
  1. A docstring, module-level comment, or header comment in the first 20 lines
  2. A `# Description:` or `"""..."""` block
  3. If none found, infer a brief description from the filename and any argparse/usage text

Keep descriptions to one sentence.

### 4. Generate the reference file

Write the output file in this exact format:

```markdown
# Script References

> Auto-generated by `/scripts-reference` â€” {date}
> Source directory: `{absolute_path_to_scripts_dir}`

| Script | Namespace | Description | Location |
|--------|-----------|-------------|----------|
| `run_alignment.sh` | `pipeline/alignment` | Runs BWA-MEM2 alignment against the reference genome | [run_alignment.sh](file://{absolute_path}) |
| `call_variants.py` | `pipeline/variants` | Calls SNPs and indels using GATK HaplotypeCaller | [call_variants.py](file://{absolute_path}) |
```

Rules for the table:
- Sort rows by namespace first, then by script name alphabetically
- Use `file://` URI scheme for the Location links â€” this makes them clickable in most editors (VS Code, Notepad++, JetBrains, etc.)
- All paths must be absolute
- Namespace `.` means the script is in the root of the scanned directory

### 5. Report

After writing the file, tell the user:
- Where the file was written (absolute path)
- How many scripts were cataloged
- Any files that were skipped and why (only mention if non-obvious skips occurred)

## Does NOT do

- Does not catalog test scripts, dev-only helpers, or CI configs
- Does not execute or validate any scripts
- Does not modify any scripts â€” this is read-only
- Does not create a `/handoff` boundary or track tasks
